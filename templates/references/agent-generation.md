# Agent Generation

## Step 1: Customize Agent TOMLs

For each agent template in [agents/](../agents/):

1. Read the template
2. Replace placeholder values (`{{...}}`) with discovery results:
   - `{{required_mcps}}` → MCPs marked required for this agent's affinity (from registry)
   - `{{optional_mcps}}` → MCPs marked optional/recommended for this agent's affinity
   - `{{fallback_chain}}` → documentation sources from dominion.toml [documentation.sources], ordered by priority
   - `{{pre_commit_commands}}` → from style.toml per-language pre_commit arrays
   - `{{file_ownership}}` → empty for now (populated by Architect during planning)
   - `{{hard_stops}}` → from dominion.toml [project.constraints.hard_rules] + defaults
   - `{{testing_styles}}` → detected from test frameworks (property-based if proptest/hypothesis found, e2e if playwright/cypress found, etc.)
   - `{{security_scope}}` → language-specific security concerns (from security-auditor.toml template)
3. Write customized TOML to `.dominion/agents/{role}.toml`

## Step 2: Render Agent .md Files

For each customized `.dominion/agents/{role}.toml`:

### Header
```
<!-- Generated by Dominion from .dominion/agents/{role}.toml — scaffold sections will be overwritten on regeneration. To customize, ask the Advisor or edit the free-form sections. -->
```

### Frontmatter (from [agent] section)
```yaml
---
name: {agent.name}
model: {agent.model}
color: {agent.color}
description: {agent.purpose}
---
```

### Rigid Scaffold Sections (deterministic from TOML)

**Startup Sequence** — from [tools.mcps]:
```
## Startup Sequence
1. Load required MCP tools: {for each required MCP, generate ToolSearch call}
2. Load optional MCP tools (if available): {for each optional MCP}
3. Read current state: `dominion-tools state position --json`
4. Read task assignment (if in execution): `dominion-tools plan task {task-id} --json`
```

**Tool Routing** — from [tools.documentation] + [tools.cli]:
```
## Tool Routing
### Documentation Lookup
Follow this chain IN ORDER. Do not skip steps:
{for each source in fallback chain, numbered}
Terminal: {fallback.action} — {fallback.message}

### CLI Commands Available
{list from [tools.cli.commands]}
```

**Governance** — from [governance]:
```
## Governance Rules
{if architectural_decisions = "stop-and-report":}
- STOP and report if you encounter an architectural decision
{for each hard_stop:}
- HARD STOP: {description}
- NEVER edit `.dominion/` TOML files directly. All data reads and writes go through `dominion-tools` commands.

### File Ownership
{if file_ownership is non-empty: list owned directories}
{if empty: "No exclusive file ownership assigned yet."}
```

**Workflow** — from [workflow]:
```
## Workflow
- Commit style: {commit_style}
- Pre-commit checks: {pre_commit commands}
- Produces: {produces}
```

### Free-Form Sections (synthesized at init)

Use ultrathink to synthesize these sections based on:
- The agent's role and purpose
- The detected project domain (from discovery)
- The project's language/framework stack
- The style.toml conventions

```
## Role & Behavioral Instructions
<!-- GENERATED AT INIT — preserved on regeneration, updated during agent evolution -->

{Synthesized instructions specific to this agent's role in THIS project.
 Example for Developer in a Rust project:
 "You are a Developer agent working on a multi-crate Rust workspace.
  Follow the project's error handling conventions: thiserror for library crates,
  anyhow for binary crates. Use tokio for all async code..."}

## Domain Context
<!-- UPDATED DURING AGENT EVOLUTION — diff shown for approval -->

{Initially minimal. Grows as the project evolves.
 Example: "This project uses a custom networking protocol. See .dominion/knowledge/protocol.md
  for wire format details."}
```

Write each rendered .md to `.claude/agents/{role}.md`.
