# Config Generation (v0.3.0)

Generate `.dominion/` directory tree and config.toml from detection + interview.

## Step 1: Directory Structure

```bash
mkdir -p .dominion/agents
mkdir -p .dominion/heuristics
mkdir -p .dominion/knowledge
mkdir -p .dominion/phases
```

## Step 2: config.toml

Generate `.dominion/config.toml` — single merged config file:

```toml
[project]
name = "{{project_name}}"
languages = [{{detected_languages}}]
frameworks = [{{detected_frameworks}}]
git_platform = "{{detected_platform}}"
test_command = "{{detected_test_command}}"

[direction]
mode = "{{from_interview}}"  # maintain | improve | restructure
# If restructure, also include:
# target_state = "{{from_interview}}"
# migration_strategy = "{{from_interview}}"  # strangler-fig | big-bang | incremental

[tools]
available = [{{detected_mcps}}]

[agents]
active = ["researcher", "architect", "developer", "reviewer", "security-auditor", "analyst", "innovation-engineer"]

[style]
testing = "{{from_interview}}"
conventions = [{{from_interview + detected conventions}}]

[auto]
halt_on_severity = "critical"
max_retries = 3
max_iterations = 10
```

Detect test command:
- Python: `uv run pytest` or `pytest` (check pyproject.toml/setup.cfg)
- Node: `npm test` (check package.json scripts.test)
- Rust: `cargo test`
- Go: `go test ./...`
- Fallback: leave empty

## Step 3: Knowledge index

Write `.dominion/knowledge/index.toml`:
```toml
# Knowledge index — populated by save_knowledge during pipeline runs
```

If direction.mode is "improve" or "restructure": create a knowledge entry with the user's pain points and direction detail so every agent sees it:

- Write `.dominion/knowledge/project-direction.md`:
  ```markdown
  # Project Direction: {Improve|Restructure}

  {if restructure: "**Target state:** {target_state from interview}"}
  {if restructure: "**Migration strategy:** {migration_strategy}"}

  ## What Needs Work
  {pain points / weak areas from interview — include as much detail as the user provided}
  ```
- Add entry to `index.toml`:
  ```toml
  [[entries]]
  topic = "project-direction"
  summary = "{if improve: 'Improvement priorities: {first pain point}'}{if restructure: 'Restructuring toward {target_state}'}"
  tags = ["research", "plan", "execute", "review"]
  path = "project-direction.md"
  referenced_files = []
  ```

## Step 7: Generate CLAUDE.md

Follow the CLAUDE.md template below. Present each section to user for approval.

```markdown
<!-- Generated by Dominion — human-owned after init. Edit freely. -->
# Project Instructions

## Dominion
This project uses Dominion for AI development.
MCP tools provide project state and context. Run `/dominion:orchestrate` for pipeline.

## Project
{vision — from interview Q1, includes what it is and who uses it}
Languages: {languages}
Frameworks: {frameworks}

## Direction
{if maintain: "Maintain — preserve existing patterns. Be conservative with changes. Match the style and structure of surrounding code."}
{if improve: "Improve — follow existing patterns but fix anti-patterns when touching code. Incremental improvement, not wholesale rewrite."}
{if restructure: "Restructure — target: {target_state}. Strategy: {migration_strategy}. New code follows target patterns, not legacy patterns. See knowledge entry 'project-direction' for full details."}

## Experience Level
{if beginner: "Beginner — explain decisions, show examples, link to docs when relevant."}
{if intermediate: "Intermediate — balanced detail. Explain non-obvious choices."}
{if advanced: "Advanced — be terse, skip obvious explanations. Focus on what's non-trivial."}

## Conventions
{detected formatter, linter, error handling}
Testing: {tdd | test-after}
{if taste.dos: "ALWAYS:"}
{for each do: "- {do}"}
{if taste.donts: "NEVER:"}
{for each dont: "- {dont}"}

## Git Conventions
{branch naming, commit format, merge strategy from detection}

## Quality Gates
Before committing, run:
{test_command from config}

## Dependencies
Package manager: {detected}
Install: `{install_command}`
Add: `{add_command}`
Run: `{run_command}`
NEVER: {prohibited commands}

## Dominion Agents
Agent definitions in `.claude/agents/`. When spawning agents with the Agent tool, pass the role
name as subagent_type (e.g., "developer", "reviewer"). These resolve to local agent files
with pipeline-specific instructions and correct model assignments.
Do NOT substitute plugin agents — Dominion agents are purpose-built for this pipeline.

## Decision Framework
- **Decide autonomously:** formatting, variable naming, test structure
- **Flag and continue:** minor deviations, non-blocking issues
- **STOP and ask:** architectural decisions, new dependencies, security concerns
```

## Step 8: Generate .mcp.json

If `.mcp.json` exists, MERGE the dominion entry — never overwrite other servers.

```json
{
  "mcpServers": {
    "dominion": {
      "command": "dominion-mcp",
      "args": []
    }
  }
}
```

## Step 9: Generate settings + hooks

Follow [settings-generation.md](settings-generation.md) and [hooks-generation.md](hooks-generation.md).

## Step 10: VSCode worktree exclusions (if applicable)

If `.vscode/` exists OR the project uses VSCode (check for `.vscode/settings.json`):

Read `.vscode/settings.json` if it exists, merge these keys (never overwrite existing values):

```json
{
  "files.watcherExclude": { "**/.claude/worktrees/**": true },
  "files.exclude": { "**/.claude/worktrees/**": true },
  "search.exclude": { "**/.claude/worktrees/**": true }
}
```

If `.vscode/settings.json` doesn't exist but `.vscode/` does, create it with just these entries.
If `.vscode/` doesn't exist, skip this step.

This prevents VSCode from tracking thousands of worktree files during pipeline execution.

Check: `command -v dominion-mcp`. If not installed: `uv tool install {plugin_mcp_path}`.
