# Generation

Generate all Dominion artifacts from detection results and interview answers.

## Step 1: Directory Structure

Create the `.dominion/` directory tree:
```bash
mkdir -p .dominion/agents
mkdir -p .dominion/knowledge
mkdir -p .dominion/memory
mkdir -p .dominion/signals
mkdir -p .dominion/phases
```

## Step 2: Project Config

Generate these files from detection results + interview answers:
- `.dominion/dominion.toml` — from [dominion.toml](../../../templates/schemas/dominion.toml), filled with project data
- `.dominion/style.toml` — from [style.toml](../../../templates/schemas/style.toml), filled with convention choices
- `.dominion/roadmap.toml` — from [roadmap.toml](../../../templates/schemas/roadmap.toml), filled with user's roadmap
- `.dominion/user-profile.toml` — from [user-profile.toml](../../../templates/schemas/user-profile.toml), filled with experience_level from interview
- `.dominion/knowledge/index.toml` — initial index with structural-awareness entry
- `.dominion/knowledge/structural-awareness.md` — copy from [structural-awareness.md](../data/structural-awareness.md)

Verify each TOML parses: `python3 -c "import tomllib; tomllib.load(open('{file}','rb'))"`

FAIL if any file is missing or unparseable. Do not proceed.

## Step 3: Generate Agents

Follow [agent-generation.md](agent-generation.md):
1. Customize agent TOMLs from templates → write to `.dominion/agents/`
2. Render agent .md files → write to `.claude/agents/`

Verify:
- Glob: `.dominion/agents/*/agent.toml` — expect at least 9 files (core agents including Security Auditor)
- Glob: `.claude/agents/*.md` — expect matching count
- Each TOML parses cleanly

FAIL if count mismatch or any unparseable.

## Step 4: Generate AGENTS.md

Generate `AGENTS.md` at project root with thin 3-line dispatch entries.

Each agent entry follows this format:
```markdown
### {Agent Name}
- Role: {one-line purpose}
- Dispatch: `mcp__dominion__step_dispatch(step: "{step}", agent: "{role}")`
- Model: {model}
```

Include the standard header:
```markdown
<!-- Auto-generated by Dominion — do not edit manually. -->
<!-- Regenerate: mcp__dominion__agents_generate() -->
# Agent Team

Generated: {ISO 8601 timestamp}

## Active Roster

{thin entries for all agents}

## Governance

**Hard stops — agents STOP and report:**
- Architectural decisions → human owns architecture
- Unapproved dependencies → check registry
- Wire format / stored data format changes → affects interoperability
- Plan deviations without documentation → plan is the contract

**Escalation:** Agent stops → reports what spec says vs what's needed → orchestrator evaluates → human decides.
```

## Step 5: Generate CLAUDE.md

Use ultrathink. Draft a lean CLAUDE.md (60-100 lines):

```markdown
<!-- Generated by Dominion — human-owned after init. Edit freely. -->
# Project Instructions

## Dominion
This project uses Dominion for AI development methodology.
MCP tools (`mcp__dominion__*`) provide all project state and methodology.
Run `/effort high` before `/dominion:orchestrate` for best results.

## Project
{name} — {vision}
Languages: {detected languages}
Direction: {direction mode}

## {Language} Conventions
{from style.toml — formatter, linter, error handling, pre-commit}

## Git Conventions
{from interview — branch naming, commit format, merge strategy}

## Quality Gates
Before committing, run:
{pre_commit commands from style.toml}

## Dependencies
Package manager: {dev_profile.name}
Install: `{dev_profile.install}`
Add dependency: `{dev_profile.add}`
Run: `{dev_profile.run}`
NEVER: {dev_profile.prohibited — list each prohibited command}
{if dev_profile.venv_required: "Virtual environment at {dev_profile.venv_path} — always use `{dev_profile.run}` prefix"}

## Decision Framework
- **Decide autonomously:** formatting, variable naming, test structure
- **Flag and continue:** minor deviations from plan, non-blocking issues
- **STOP and ask:** architectural decisions, new dependencies, wire format changes, security concerns
```

No governance rules, no tool routing, no pipeline dispatch in CLAUDE.md. All served dynamically by MCP.

Present each section to the user for approval/edit/skip. Write final CLAUDE.md to project root.

## Step 6: Generate .mcp.json

If `.mcp.json` exists at project root, read it and preserve existing entries.
If it does not exist, create it.

Add the Dominion MCP server entry:
```json
{
  "mcpServers": {
    "dominion": {
      "command": "dominion-mcp"
    }
  }
}
```

Merge with existing entries — never overwrite other MCP servers.

## Step 7: Generate settings.local.json

Follow [settings-generation.md](settings-generation.md):
- Extend `.claude/settings.local.json` with Dominion permissions and native hooks
- Add MCP tool permissions:
  ```json
  {
    "permissions": {
      "allow": [
        "mcp__dominion__*",
        "Read(~/.claude/plugins/cache/dominion/**)"
      ]
    }
  }
  ```
- Auto-derive deny rules from dev profile prohibited commands:
  ```json
  {
    "permissions": {
      "deny": [
        "Bash(uv pip install*)",
        "Bash(pip install*)"
      ]
    }
  }
  ```
  For each entry in `dev_profile.prohibited`, add `"Bash({command}*)"` to the deny list.
- Configure serena project activation with detected LSPs

## Step 8: Generate Hooks

Follow [hooks-generation.md](hooks-generation.md):
- Create hookify governance rules (source-diving prevention, session-end checkpoint)
- Native hooks for blocker warning and session-start resume

Create governance hook scripts in `.claude/hooks/`:

### block-dominion-access.sh
Prevents agents from directly reading `.dominion/` files (must use MCP tools):
```bash
#!/bin/sh
# Agents must use mcp__dominion__* tools, not direct file access
input="$(cat)"
file_path=$(echo "$input" | python3 -c "import sys,json; print(json.load(sys.stdin).get('file_path',''))" 2>/dev/null)
case "$file_path" in
  */.dominion/*) printf '{"decision":"deny","reason":"Use mcp__dominion__* tools to access .dominion/ data"}\n'; exit 0 ;;
esac
exit 0
```

### verify-agent-submitted.sh
Verifies agent outputs are submitted through MCP:
```bash
#!/bin/sh
# Placeholder — expanded by dominion-mcp at runtime
exit 0
```

Make all hook scripts executable: `chmod +x .claude/hooks/*.sh`

## Step 9: Install dominion-mcp

Check if `dominion-mcp` is already installed:
```bash
command -v dominion-mcp
```

If not installed:
1. Determine the plugin root directory (derive from this skill's base directory)
2. Check `uv` is available: `command -v uv`
3. Install: `uv tool install dominion-mcp` (from plugin distribution)
4. Verify: `dominion-mcp --version`

If install fails, warn but do not block — user can install manually after onboard.

## Step 12: Generate DOMINION.md

Generate `DOMINION.md` at project root from [dominion-md.md](../../../templates/dominion-md.md):
1. Read `dominion.toml` — extract project info, direction, workflow config
2. Read `roadmap.toml` — extract milestone and phase summary
3. Read all `.dominion/agents/*/agent.toml` — build agent roster table
4. Fill template placeholders
5. Write to `DOMINION.md` at project root
